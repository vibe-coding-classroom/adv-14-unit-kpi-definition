以下是在 **GitHub Classroom** 部署其作業 (Assignments) 的具體建議：

### 1. 範本倉庫 (Template Repo) 配置建議
性能評鑑需要標準化的分析模板與監測工具，建議範本包含：
*   **📂 `src/telemetry_visualizer.js`**：前端指標計算模組。預留 `calculateRTT()` 與 `updateTrafficLight()` 的介面，讓學員實作延遲監測與視覺警示。
*   **📂 `docs/BENCHMARK_REPORT.md`**：正式基準測試報告。提供表格模板，包含：測試環境（距離/障礙物）、樣本數、平均延遲、標準差 (Jitter) 等欄位。
*   **📂 `tests/latency_emulator.js`**：延遲模擬工具。這是一個輔助腳本，能人為注入 200ms 的延遲，用來測試學員寫的「效能警示 UI」是否能正確變色。
*   **📂 `SPEC_SHEET.md`**：系統性能規格書。學員最終需在此填入車輛的「官方參數」（例如：最大穩定連線距離、RTT 通訊極限值）。

### 2. 作業任務部署細節

#### 任務 1：專業級數位 KPI 儀表板 (Metric-Driven Dashboard Lab)
*   **目標**：建立系統透明度，讓操控者能在連線惡化前獲得視覺預警。
*   **Classroom 部署建議**：
    *   **RTT 監測實作**：學員需在前端 JavaScript 透過 `Ping/Pong` 時戳差計算實時延遲。
    *   **自適應狀態感知**：實作 UI 變色邏輯。綠色 (< 80ms)、黃色 (80-120ms)、紅色 (> 120ms)。
    *   **PR 截圖驗證**：提交 PR 時需附上不同延遲狀態下的 UI 截圖，證明監測邏輯與視覺回饋同步。

#### 任務 2：系統性能規格書極限測試 (System Spec Benchmarking Lab)
*   **目標**：執行科學實驗，定義「控制失效」的臨界數據點。
*   **Classroom 部署建議**：
    *   **多場景取樣**：學員需在「室內空曠」、「隔牆連線」與「長距離」三種場景下各取樣至少 200 筆通訊數據。
    *   **臨界點診斷**：找出當 RTT 到達多少 ms 時，PID 循跡會開始產生不可控的震盪，這就是該系統的「安全極限點」。
    *   **規格書彙編**：在 README 中正式載錄你的「Vibe-Car 性能規格」，這對於工業產品推廣至關重要。

#### 任務 3：優化收益對比挑戰 (Performance Delta Benchmark Lab)
*   **目標**：計算「工程改良」帶來的實質投資報酬率 (ROI)。
*   **Classroom 部署建議**：
    *   **優化幅度計算**：學員實施一項優化（如：改用二進位封包），計算優化前後 KPI 的真實提升比率 ($\Delta \%$)。
    *   **控制平穩度反思**：說明延遲降低後，對車體「高速過彎」時的側向加速度穩定度有何實質幫助。
    *   **Push Final**：提交具備「數據支撐優化決策」的最終整合報告。

### 3. 性能工程師導師點評標準 (KPI & Quality Standards)
此單元的價值在於 **「對系統行為的數字掌控力與可靠性保證」**：
*   [ ] **測量嚴謹性**：性能報告是否包含樣本分佈分析，而非僅僅給出一個平均值？
*   [ ] **邊界定義能力**：是否能清楚指出該控制系統在哪種網路條件下「不推薦使用」？（展現風險評估意識）。
*   [ ] **視覺導航力**：前端儀表板的 KPI 顯示是否直觀，能讓一般操作者快速理解系統健康度？
